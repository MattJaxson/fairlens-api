Analysis Endpoints — /api/v1/analyze
POST /api/v1/analyze/text All Tiers
Analyze text for bias across gender, race, age, and disability categories. Returns per-category scores (0-100), flagged phrases with suggestions, and overall fairness score.
Example Request & Response
{ "text": "We need a young, aggressive salesman...", "categories": ["gender", "race", "age", "disability"] } // Response { "report_id": "abc-123", "overall_score": 34.2, "bias_scores": [ {"category": "gender", "score": 28.5, "flagged_phrases": [...]}, {"category": "age", "score": 40.0, "flagged_phrases": [...]} ], "recommendations": ["Replace 'salesman' with 'salesperson'"] }
POST /api/v1/analyze/dataset All Tiers
Analyze a dataset for fairness metrics: disparate impact ratio, statistical parity difference, equal opportunity difference. Pass data as a list of JSON objects.
GET /api/v1/analyze/report/{report_id} All Tiers
Retrieve a previously generated analysis report by ID.

POST /api/v1/fairness/audit All Tiers
Racial fairness audit on JSON data. Returns disparate impact ratios per group, findings, and recommendations against EEOC 4/5ths rule or community-defined threshold.

Upload a CSV file for racial fairness audit.

Fairness-aware sample reweighting. Returns original and reweighted outcome rates per group.
POST /api/v1/fairness/reweight/csv Pro+
Upload a CSV file for fairness-aware reweighting.

Cryptographic Provenance Ledger (CPL)

Every community fairness standard created through CDF v1.0 is entered into the Cryptographic Provenance Ledger — a SHA-256 hash chain that makes the entire audit trail tamper-evident.

When a CommunityConfig is created, Libra computes a SHA-256 hash of its contents and links it to the previous config’s hash. This forms an append-only chain: if any historical record is modified, every subsequent hash breaks. Regulatory reports include the hash for independent verification.

For compliance officers: The CPL proves your fairness standard wasn’t modified after the fact. Present the hash at audit. Regulators can independently verify the chain without access to your data.

HOW IT WORKS
1
Community sets standard via POST /config
2
Libra computes SHA-256 hash & chains to predecessor
3
Hash included in all compliance reports for verification
POST /api/v1/fairness/community/config Enterprise
Creates config and enters it into the provenance ledger with SHA-256 hash chain.
Q2Q Translation Matrix

The Qualitative-to-Quantitative Translation Matrix documents exactly how community input maps to fairness parameters. When residents say “We want equitable lending outcomes,” the Q2Q records how that becomes θ=0.85, priority_groups=["Black", "Latinx"].

Every parameter in a community fairness standard is traceable through the Q2Q back to the community’s own words. The matrix is stored in config provenance and surfaced in compliance reports.

For compliance officers: The Q2Q closes the gap regulators care about most — proving your fairness threshold wasn’t arbitrary. It documents the deliberative process that produced each quantitative parameter.

TRANSLATION EXAMPLE
QUALITATIVE INPUT
“80% isn’t good enough for our community. We want at least 90% parity.”
→
QUANTITATIVE PARAMETER
threshold: 0.90
metric: disparate_impact
POST /api/v1/fairness/community/config Enterprise

GET /api/v1/fairness/community/config Pro+
Retrieve the active config including its Q2Q translation mappings.
